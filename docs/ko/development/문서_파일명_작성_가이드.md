# 문서 파일명 작성 가이드

## 개요

본 가이드는 프로젝트 내 모든 문서 파일의 파일명을 한글로 작성하는 표준 규칙을 정의합니다. 이를 통해 문서의 가독성과 접근성을 향상시키고, 한국어 사용자에게 더 직관적인 문서 구조를 제공합니다.

## 기본 원칙

### 1. 한글 파일명 사용 필수
- **모든 문서 파일명은 한글로 작성**합니다.
- 영문 파일명은 사용하지 않습니다.
- 기존 영문 파일명 문서는 점진적으로 한글 파일명으로 변경합니다.

### 2. 파일명 구성 규칙

#### 2.1 기본 구조
```
주제_세부내용_문서유형.md
```

#### 2.2 구체적인 예시
- ✅ **올바른 예시**:
  - `구글클라우드_SpaceONE_웹훅_연동_가이드.md`
  - `개발_환경_설정_가이드.md`
  - `API_사용법_매뉴얼.md`
  - `데이터베이스_스키마_설계서.md`
  - `보안_정책_문서.md`

- ❌ **잘못된 예시**:
  - `google-cloud-spaceone-webhook-integration.md`
  - `development-environment-guide.md`
  - `api-usage-manual.md`

#### 2.3 언더스코어(_) 사용 규칙
- 단어 간 구분은 **언더스코어(_)**를 사용합니다.
- 하이픈(-), 공백( ), 점(.) 등은 사용하지 않습니다.
- 카멜케이스(camelCase)나 파스칼케이스(PascalCase)는 사용하지 않습니다.

### 3. 문서 유형별 명명 규칙

#### 3.1 가이드 문서
```
[주제]_[세부내용]_가이드.md
```
- 예시: `웹훅_연동_설정_가이드.md`, `개발_환경_구축_가이드.md`

#### 3.2 매뉴얼 문서
```
[시스템명]_[기능]_매뉴얼.md
```
- 예시: `SpaceONE_API_사용_매뉴얼.md`, `모니터링_시스템_운영_매뉴얼.md`

#### 3.3 설계서/명세서
```
[대상]_[유형]_설계서.md 또는 [대상]_[유형]_명세서.md
```
- 예시: `데이터베이스_스키마_설계서.md`, `API_인터페이스_명세서.md`

#### 3.4 정책/규칙 문서
```
[분야]_정책_문서.md 또는 [분야]_규칙_문서.md
```
- 예시: `보안_정책_문서.md`, `코딩_규칙_문서.md`

#### 3.5 분석/리포트 문서
```
[대상]_[유형]_분석서.md 또는 [대상]_[유형]_리포트.md
```
- 예시: `성능_테스트_분석서.md`, `사용자_피드백_리포트.md`

### 4. 특수 상황 처리

#### 4.1 영문 고유명사 포함
- 영문 고유명사(제품명, 서비스명 등)는 그대로 유지합니다.
- 예시: `AWS_Lambda_배포_가이드.md`, `Docker_컨테이너_운영_매뉴얼.md`

#### 4.2 버전 정보 포함
```
[주제]_[세부내용]_v[버전]_[문서유형].md
```
- 예시: `API_명세서_v2.1_문서.md`, `시스템_아키텍처_v1.0_설계서.md`

#### 4.3 날짜 정보 포함 (필요시)
```
[주제]_[세부내용]_[YYYY-MM-DD]_[문서유형].md
```
- 예시: `월간_성능_리포트_2024-01-31_분석서.md`

### 5. 디렉토리별 파일명 규칙

#### 5.1 기술 문서 (`docs/ko/technical/`)
- 기술적 구현, 연동, 설정 관련 문서
- 예시: `구글클라우드_SpaceONE_웹훅_연동_가이드.md`

#### 5.2 개발 문서 (`docs/ko/development/`)
- 개발 환경, 코딩 규칙, 개발 프로세스 관련 문서
- 예시: `개발_환경_가이드.md`, `코드_품질_가이드.md`

#### 5.3 운영 문서 (`docs/ko/operations/`)
- 시스템 운영, 배포, 모니터링 관련 문서
- 예시: `시스템_배포_절차서.md`, `장애_대응_매뉴얼.md`

#### 5.4 사용자 가이드 (`docs/ko/user-guide/`)
- 최종 사용자를 위한 사용법, 튜토리얼 문서
- 예시: `사용자_시작_가이드.md`, `기능_사용법_매뉴얼.md`

## 파일명 변경 절차

### 1. 기존 영문 파일명 문서 처리
1. **새로운 한글 파일명으로 문서 생성**
2. **기존 영문 파일명 문서 삭제**
3. **관련 링크 및 참조 업데이트**

### 2. 변경 시 주의사항
- Git 히스토리 보존을 위해 `git mv` 명령어 사용 권장
- 다른 문서에서 참조하는 링크가 있는지 확인
- README.md 파일의 목차나 링크 업데이트

### 3. 변경 예시
```bash
# 기존 파일 삭제 및 새 파일 생성
rm docs/ko/technical/google-cloud-spaceone-webhook-integration.md
# 새로운 한글 파일명으로 문서 작성
# docs/ko/technical/구글클라우드_SpaceONE_웹훅_연동_가이드.md
```

## 참고사항

### 1. 파일 시스템 호환성
- 대부분의 현대 파일 시스템은 한글 파일명을 지원합니다.
- Windows, macOS, Linux 모두 UTF-8 인코딩 한글 파일명 지원

### 2. Git 버전 관리
- Git은 UTF-8 인코딩 한글 파일명을 완전히 지원합니다.
- 파일명 변경 시 Git 히스토리가 유지됩니다.

### 3. 웹 서버 호환성
- 현대적인 웹 서버들은 UTF-8 인코딩 한글 파일명을 지원합니다.
- 필요시 URL 인코딩을 통해 웹 접근 가능

## 결론

한글 파일명 사용을 통해 문서의 직관성과 접근성을 크게 향상시킬 수 있습니다. 모든 팀원은 본 가이드를 준수하여 일관성 있는 문서 관리 체계를 유지해야 합니다.

새로운 문서 작성 시에는 반드시 한글 파일명을 사용하고, 기존 영문 파일명 문서는 점진적으로 한글 파일명으로 변경하여 프로젝트 전체의 문서 품질을 향상시켜 나가겠습니다.
